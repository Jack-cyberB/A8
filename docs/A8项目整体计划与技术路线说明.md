# A8项目整体计划与技术路线说明

## 1. 这份文档是做什么的

这份文档是给项目负责人自己看的总说明，目的是把下面几件事讲清楚：

- 赛题真正要求我们做什么
- 当前项目已经做到了什么
- 当前技术实现和最初设想有什么差异
- `V0 / V1 / V2` 分别代表什么
- 接下来应该如何从“可演示原型”走向“更像正式作品”

这份文档不追求学术表达，而是追求帮助后续开发时不再混乱。

---

## 2. 根据赛题和答疑，我对项目目标的理解

结合赛题文档、指导手册和直播答疑，A8 本质上不是一个单纯的“问答系统”，而是一个：

`建筑能源数据管理 + 查询统计分析 + 智慧运维辅助 + 大模型能力集成`

的一体化系统。

### 2.1 赛题最核心的几件事

1. 要有**自主构建或整理的能耗数据集**
2. 要能做**多条件查询和统计分析**
3. 要有**可视化展示**
4. 要把**领域知识和大模型能力结合起来**
5. 要最终形成一个**界面可操作、流程可演示**的系统

### 2.2 直播答疑里最重要的信息

根据直播答疑，下面这些点非常关键：

- 不强制接真实物理设备，导入数据即可
- 可以直接使用开源数据集，也可以用自己已有数据
- 数据不一定非要全，先做自己能做好的参数
- 不要求必须做完整权限系统
- 不要求必须做 APP，网站即可
- 不要求必须全部本地模型，允许调用在线 API
- 系统主要面向 `to B` 运维人员，也可以兼顾普通师生这类 `to C` 查询场景
- 响应速度以演示级为主，尽量控制在 `30 秒内`
- 可以只选部分数据参数做深，不必一开始全做

### 2.3 赛题对“能耗分析”并不只等于电能

这一点很重要。赛题希望的是建筑能源管理，不是“只做电表曲线”。  
电能可以作为第一阶段切入点，但后续应逐步扩展到：

- 水
- 空调系统相关参数
- 冷热源相关参数
- 环境影响因素
- 设备运行状态

所以当前只做电能，是合理起步，但不是最终目标。

## 3. 你最初设想的技术路线

你一开始的设想是：

- 前端：`Vue`
- 后端主服务：`Spring Boot`
- 数据库：`MySQL`
- AI 层：`Python`
- 知识库 / RAG：`RAGFlow`

这是一个比较典型、也比较正式的企业式技术方案。它的优点是：

- 分层明确
- 后期可扩展性更强
- 更接近生产系统
- 适合后续多人协作和模块拆分

如果项目时间充足，最终往这个方向收敛是对的。

## 4. 当前项目实际采用的技术架构

当前仓库里已经落地的架构不是你最初那套正式架构，而是一套**为了快速交付 V0/V1/V2 原型而做的轻量实现**。

### 4.1 当前实际技术栈

- 前端：`Vue 3 + Element Plus + ECharts`
- 后端：`Python 原生 HTTP Server`
- 数据存储：
  - 原始数据：`CSV / JSON / JSONL`
  - 运行数据：`jsonl` 文件
- AI 接入：
  - 模板诊断
  - DeepSeek API 直连
  - 失败自动降级
- 测试：
  - Python 单测
  - API smoke
  - Playwright E2E

### 4.2 当前目录对应关系

- `frontend/`：页面和交互
- `backend/`：接口、规则、AI 调用、闭环逻辑
- `data/raw/`：原始数据
- `data/normalized/`：标准化数据
- `data/runtime/`：运行过程生成的日志与状态
- `tests/e2e/`：前端回归测试
- `scripts/`：数据处理和回归脚本

### 4.3 这套轻量架构为什么会出现

原因不是方向错了，而是为了先解决“单人、时间有限、先跑通演示闭环”的问题。

也就是说：

- 它不是最终企业版架构
- 它是一个**为了快速验证功能可行性**而做的中间阶段架构

## 5. 当前实现和最初设想的差异

下面把差异直接列出来。

### 5.1 后端差异

你原本想要：

- `Spring Boot` 做主服务
- `MySQL` 做结构化存储
- `Python` 单独负责 AI 层

当前实际是：

- 整个后端几乎都在 `Python` 里
- 没有 Spring Boot
- 没有 MySQL
- 规则检测、接口服务、AI 调用、运行日志都在 Python 层里

#### 影响

优点：

- 开发快
- 修改快
- 适合比赛冲刺

缺点：

- 单文件/轻服务结构不够正式
- 后期维护压力会变大
- 不够贴近你原本想做的“标准分层架构”

### 5.2 数据层差异

你原本想要：

- `MySQL` 存业务数据
- 结构化管理能耗、告警、知识库、操作记录

当前实际是：

- 标准化能耗数据仍是文件型数据
- 运行态记录使用 `jsonl`
- 没有真正的数据库建模

#### 影响

优点：

- 上手快
- 不需要先做数据库设计

缺点：

- 查询能力有限
- 后期统计、筛选、关联扩展会越来越麻烦
- 不利于演示“平台化、工程化”能力

### 5.3 AI 层差异

你原本想要：

- `Python` 负责 AI 能力
- `RAGFlow` 搭建知识库

当前实际是：

- Python 确实承担了 AI 调用
- 但还没有真正接入 `RAGFlow`
- 当前更多是“轻量知识片段 + 模板诊断 + LLM 直连”

#### 影响

优点：

- 已经有 AI 入口
- 能先跑通问答与诊断流程

缺点：

- 还不算真正完整的 RAG 方案
- 知识检索的完整度和可解释性还不够强

### 5.4 前端差异

你原本期望的是比较完整、成熟的系统界面。  
当前实际效果更偏向：

- 功能能点通
- 图表能展示
- 闭环能跑通
- 但视觉层和产品层仍比较粗糙

你的主观感受“只感受到粗糙前端图表显示，简单 AI 问答，完成度低”，这个判断是成立的。  
当前项目在“功能连通”上已经比表面看到的多很多，但在“使用体验”和“作品完成度”上，确实还不够。

## 6. V0 / V1 / V2 到底分别改了什么

为了避免之后再混淆，这里统一定义：

### 6.1 V0：最小可演示原型

目标：

- 把系统先跑起来
- 页面先能看
- 用测试数据把核心链路连通

主要内容：

- 四个页面骨架
- 基础图表
- 异常列表
- 简单 AI 诊断接口
- 本地 mock / 规则逻辑

这时重点是“有东西能演示”，不是“做得精细”。

### 6.2 V1：闭环版

目标：

- 不只是展示数据，而是形成业务闭环

主要增加：

- 异常状态流转
- `ack / ignore / resolve`
- 异常详情
- 历史处理记录
- AI 诊断写入处理草稿
- 更多真实数据接入
- 回归测试基线

这时重点从“页面展示”升级为“业务流程可走通”。

### 6.3 V2：交付版 / 强化版

目标：

- 从“能演示”向“像一套完整作品”靠近

主要增加：

- 复盘记录
- 导出 CSV
- AI 评估
- 系统健康检查
- 诊断质量反馈
- acceptance 测试

这时重点从“闭环”升级为“可交付、可验证、可解释”。

## 7. 当前项目完成度判断

### 7.1 如果目标是“比赛原型”

当前已经达到：

- 有真实数据
- 有页面
- 有查询分析
- 有异常检测
- 有 AI 问答/诊断
- 有运维处理流程
- 有导出和复盘
- 有自动化测试

所以从“竞赛作品原型”角度，当前已经不是空壳了。

### 7.2 如果目标是“最终高完成度作品”

当前还没有达到。  
主要差距在以下几个方面：

- 分析维度还偏少
- 前端产品感不足
- AI 仍偏轻量
- 数据层还不够工程化
- 架构还没有回到你最初想要的标准分层

也就是说：

- **业务闭环已经初步成立**
- **作品完成度还远未封顶**

## 8. 当前最需要优化的点

根据赛题要求、你的使用感受和当前项目状态，后续优化重点建议如下。

### 8.1 第一优先级：扩展分析维度

当前主要围绕电能。后续要补：

- 水能耗
- 空调系统参数
- 环境参数
- 设备状态参数

建议做法：

- 在当前统一数据结构基础上扩展字段
- 不必一次全部做齐
- 可以先增加 `电 + 水` 或 `电 + 空调`

### 8.2 第二优先级：优化前端展示

当前前端问题主要不是“没有图”，而是：

- 视觉不够完整
- 图表层次感不足
- 信息密度组织不够好
- 页面状态提示比较粗糙

建议重点优化：

- 卡片排版
- 图表配色和留白
- 页面空状态/错误状态/加载状态
- 图表切页稳定性
- 文案专业感

### 8.3 第三优先级：把 AI 从“简单问答”升级成“领域助手”

当前 AI 更像一个“能返回结果的接口”。  
后续应该进一步强化：

- 数据查询型问答
- 运维流程指导
- 异常原因分析
- 知识引用更明确
- 更像领域专家而不是普通聊天

### 8.4 第四优先级：回到你原始设想的正式架构

如果后续时间允许，建议逐步收敛回你原本计划：

- 前端保留 Vue
- 后端主服务切到 `Spring Boot`
- 数据落到 `MySQL`
- AI 层独立成 `Python` 服务
- 知识库正式接入 `RAGFlow`

这个收敛过程建议渐进式，而不是重做。

## 9. 推荐的后续技术演进路线

### 阶段 A：继续在当前架构上补作品完成度

优先做：

- 多能耗类型扩展
- 前端体验优化
- AI 回答质量优化
- 文档与演示材料补齐

原因：

- 当前最缺的是作品呈现完整度
- 不是立刻换技术栈

### 阶段 B：把数据和状态持久化

建议先做：

- 用 `MySQL` 或至少 `SQLite` 替换部分 `jsonl`
- 把告警、复盘、诊断反馈结构化

原因：

- 这是从“脚本型项目”走向“平台型项目”的关键一步

### 阶段 C：主后端切回 Spring Boot

这一步的意义是：

- 更符合你最初预想
- 更方便写正式技术方案文档
- 更贴近企业系统表达方式

建议做法：

- 保持前端接口不变
- 把 Python 当前规则接口逐步迁移
- AI 部分继续由 Python 保留独立服务

### 阶段 D：接入 RAGFlow

这一步重点不是“为了用而用”，而是：

- 让知识库构建更规范
- 让知识引用更可信
- 让答辩时更容易说明“领域知识增强”的实现路径

## 10. 我对当前项目的总体判断

### 10.1 结论一句话

当前项目方向是对的，但它现在更像一个“为了冲刺比赛先做出来的可运行原型”，还不是你最初设想的那种正式、成熟、标准分层系统。

### 10.2 更具体一点

- 如果目标是 `V0/V1/V2 原型验证`：当前路线是合理的
- 如果目标是 `最终答辩作品完成度`：还需要继续打磨
- 如果目标是 `完全贴合你最初设想的正式架构`：还没有收敛到位

### 10.3 当前最现实的策略

不是立刻推翻当前项目，而是：

1. 继续把功能和展示做完整
2. 补多能耗分析能力
3. 提升前端完成度
4. 再逐步迁移到 `Spring Boot + MySQL + Python AI + RAGFlow`

这样最稳，也最符合比赛节奏。

---

## 11. 后续开发时统一遵循的原则

后续为了不再偏航，建议统一按下面原则推进：

- 先保证作品完整度，再升级正式架构
- 先做比赛最能加分的功能，再做大重构
- AI 要和业务场景绑定，不能停留在泛问答
- 数据分析不止电能，后续必须扩展
- 前端不只要“能显示”，还要“像一个完整系统”
- 所有新增能力尽量保留接口兼容，方便后续迁移到 Spring Boot
